# 校园失物招领智能匹配系统项目开发方案

## 1. 项目概述

### 1.1 项目名称

基于链表与哈希索引的校园失物招领智能匹配系统

### 1.2 开发形式

- 开发语言：C 语言
- 程序形式：控制台菜单程序
- 数据保存方式：文件输入 / 文件输出
- 开发人数：2 人
- 项目性质：数据结构课程期末大作业

### 1.3 项目背景

在校园生活中，学生经常会遗失校园卡、钥匙、雨伞、水杯、耳机、书本等物品。传统失物招领方式通常依赖人工发布和人工查找，效率较低，也容易出现信息遗漏。

本项目设计一个校园失物招领智能匹配系统，通过链表、哈希表、栈、排序和字符串匹配等数据结构与算法，实现失物信息和拾物信息的管理，并根据物品类别、名称、颜色、地点、时间和关键词自动计算匹配度，帮助用户快速找到可能对应的失物或拾物。

### 1.4 项目目标

- 实现失物信息和拾物信息的录入、查询、修改、删除。
- 使用链表作为系统核心数据存储结构。
- 支持从文件读取测试数据，并将运行结果保存到文件。
- 实现失物和拾物之间的智能匹配。
- 输出匹配分数和匹配原因。
- 实现匹配结果排序、地点统计、撤销操作等扩展功能。
- 体现数据结构课程知识在实际场景中的应用。

### 1.5 项目特色

- 区别于普通信息管理系统，重点突出“智能匹配”。
- 链表贯穿多个模块，包括失物链表、拾物链表、匹配结果链表、哈希冲突链表和撤销栈链表。
- 使用规则评分法解释匹配依据，便于演示和写报告。
- 使用文件保存测试数据，满足课程作业要求。
- 场景贴近校园生活，具有一定实际应用价值。

## 2. 需求分析

### 2.1 用户角色

本系统主要面向以下用户：

- 丢失物品的学生：录入失物信息，查询可能匹配的拾物。
- 捡到物品的学生：录入拾物信息，查询可能对应的失主。
- 系统管理员或班级负责人：维护失物和拾物信息，统计校园高发地点。

### 2.2 功能需求

系统应包含以下功能：

1. 录入失物信息
2. 录入拾物信息
3. 查看失物列表
4. 查看拾物列表
5. 修改失物或拾物信息
6. 删除失物或拾物信息
7. 按编号查询
8. 按关键词查询
9. 按地点查询
10. 按类别查询
11. 智能匹配失物与拾物
12. 查看匹配结果
13. 确认找回或归还
14. 统计校园失物高发地点
15. 撤销最近一次新增或删除操作
16. 从文件读取数据
17. 保存数据到文件
18. 退出系统

### 2.3 非功能需求

- 程序运行稳定，错误输入不会导致程序崩溃。
- 菜单提示清晰，便于课堂演示。
- 测试数据数量充足，能够体现匹配效果。
- 文件格式简单，便于人工检查和修改。
- 程序结构清晰，便于双人分工和后续维护。

## 3. 总体设计

### 3.1 系统模块划分

系统可划分为以下模块：

| 模块 | 主要功能 |
| --- | --- |
| 菜单交互模块 | 显示菜单，接收用户选择，调用对应功能 |
| 信息管理模块 | 失物和拾物信息的新增、删除、修改、查询 |
| 链表操作模块 | 链表节点创建、插入、删除、查找、遍历、释放 |
| 文件读写模块 | 从文件读取数据，将数据保存到文件 |
| 智能匹配模块 | 计算失物和拾物之间的匹配分数 |
| 匹配结果模块 | 生成、保存、排序和显示匹配结果 |
| 哈希查询模块 | 按类别或关键词建立索引，提高查询效率 |
| 撤销操作模块 | 记录最近操作，支持撤销新增或删除 |
| 统计分析模块 | 统计校园失物高发地点 |

### 3.2 系统运行流程

1. 程序启动，显示主菜单。
2. 用户选择是否从文件读取已有数据。
3. 用户通过菜单录入、查询、修改或删除失物 / 拾物信息。
4. 用户选择智能匹配功能，系统遍历失物链表和拾物链表。
5. 系统根据匹配规则计算匹配分数，并生成匹配结果链表。
6. 匹配结果按分数从高到低排序。
7. 用户查看匹配结果，并可确认找回或归还。
8. 用户可查看地点统计结果。
9. 程序退出前将数据保存到文件。

## 5. 智能匹配算法设计

### 5.1 匹配思路

系统将每一条失物记录与每一条拾物记录进行比较，根据多个条件计算匹配分数。如果匹配分数达到最低显示阈值，则将该结果加入匹配结果链表。

### 5.2 匹配评分规则

采用 100 分制规则评分：

| 匹配条件 | 分值 |
| --- | ---: |
| 物品类别一致 | 25 |
| 名称关键词相同或相似 | 25 |
| 颜色一致 | 20 |
| 地点相同或相近 | 15 |
| 丢失 / 拾取时间接近 | 10 |
| 其他关键词命中 | 5 |

### 5.3 匹配等级

| 分数范围 | 匹配等级 |
| --- | --- |
| 80 分及以上 | 高度疑似匹配 |
| 60 到 79 分 | 可能匹配 |
| 40 到 59 分 | 低可信匹配 |
| 40 分以下 | 不显示 |

### 5.4 匹配结果示例

```text
失物：蓝色校园卡，地点：图书馆二楼
拾物：蓝色学生卡，地点：图书馆门口
匹配分数：85
匹配等级：高度疑似匹配
匹配原因：类别一致 +25，颜色一致 +20，名称关键词相似 +25，地点相近 +15
```

### 5.5 匹配结果排序

匹配结果使用链表保存。生成所有匹配结果后，使用链表排序算法按照匹配分数从高到低排列。

可选择的排序方法：

- 冒泡排序：实现简单，适合课程展示。
- 插入排序：适合链表结构，代码逻辑较清晰。

建议采用插入排序，因为它更适合链表节点的重新连接。

## 7. 哈希查询设计

### 7.1 设计目的

为了体现哈希表在查询中的应用，系统可按照物品类别或关键词建立哈希索引。用户按类别或关键词查询时，系统通过哈希值快速定位对应链表，提高查询效率。

### 7.2 冲突处理

采用链地址法解决哈希冲突：

- 哈希表本质上是一个指针数组。
- 每个数组位置称为一个哈希桶。
- 哈希值相同的记录以链表形式挂接在同一个桶中。

### 7.3 查询流程

1. 用户输入类别或关键词。
2. 系统计算哈希值。
3. 根据哈希值定位哈希桶。
4. 遍历桶内链表。
5. 输出符合条件的记录。

## 8. 撤销操作设计

### 8.1 设计目的

为了提高系统可用性，支持撤销最近一次新增或删除操作。

### 8.2 实现思路

使用栈结构保存最近操作：

- 新增记录时，将操作类型和记录内容压入操作栈。
- 删除记录时，先备份被删除记录，再将操作压入操作栈。
- 执行撤销时，从栈顶弹出最近操作。

### 8.3 撤销规则

| 最近操作 | 撤销方式 |
| --- | --- |
| 新增记录 | 从对应链表中删除该记录 |
| 删除记录 | 将备份记录重新插入对应链表 |

本项目只要求撤销最近一次新增或删除操作，不要求撤销修改操作。

## 9. 地点统计设计

### 9.1 功能目标

统计失物记录中不同地点出现的次数，输出校园失物高发地点。

### 9.2 统计方式

遍历失物链表，按照地点名称累计数量。统计结果可以使用临时链表保存，每个节点记录一个地点及其出现次数。

### 9.3 输出示例

```text
校园失物高发地点统计：
1. 图书馆：6 次
2. 食堂：4 次
3. 教学楼 A 区：3 次
4. 操场：2 次
```

## 14. 复杂度分析建议

报告中可对主要功能进行复杂度分析：

| 功能 | 时间复杂度 | 说明 |
| --- | --- | --- |
| 链表插入 | O(1) 或 O(n) | 头插法为 O(1)，尾插法需遍历 |
| 链表查找 | O(n) | 需要顺序遍历链表 |
| 链表删除 | O(n) | 需要先找到目标节点 |
| 智能匹配 | O(mn) | m 为失物数量，n 为拾物数量 |
| 匹配结果排序 | O(k^2) | k 为匹配结果数量，链表插入排序或冒泡排序 |
| 哈希查询 | 平均 O(1) | 冲突较少时查询效率较高 |
| 地点统计 | O(n) | 遍历失物链表 |
| 撤销操作 | O(1) 或 O(n) | 出栈为 O(1)，撤销新增时删除节点可能为 O(n) |
